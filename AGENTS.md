# gb_server — C++ network game server framework

## Quick start

```bash
# Windows — run from "Command Prompt for VS" (VS2026 is not in PATH by default)
install_deps.bat
conan install . --build=missing
cmake --preset conan-debug
cmake --build --preset conan-debug --parallel

# Linux — same flow, but always use a profile to work around Conan bugs:
conan install . -pr=profiles/clang_debug_pr --build=missing
cmake --preset conan-debug
cmake --build --preset conan-debug --parallel

# Run server from build output
start_process.bat -t Debug -n server_test.exe
# The executable expects -t <app_type> -r <res_path> at minimum
```

## Build system quirks

- **Conan-generated presets** — `CMakePresets.json` is generated by Conan into `build/<config>/generators/`. `CMakeUserPresets.json` includes them. **Never check in generated presets.**
- **Boost 1.90 + Conan bug** — Cobalt module incompatible. Always set `boost/*:without_cobalt=True` in profile. Boost charconv needs `BOOST_CHARCONV_DISABLE_FLOAT128` to avoid float128 compile errors on clang/gcc.
- **protobuf uses DLLs on Windows** — `PROTOBUF_USE_DLLS` must be defined. CMake `cmake/base.cmake` handles DLL deployment to target output dir.
- **C++23 required** — `CMAKE_CXX_STANDARD 23`, no compiler extensions. MSVC needs `/EHa` and `/utf-8` flags.
- **Build output layout**: `build/<config>/<config>/bin/` and `build/<config>/<config>/lib/`.

## Architecture overview

```
test/server_test/main.cpp     ← entrypoint (MyApp extends App)
  └─ App (src/app/)            ← frame loop, lifecycle hooks, graceful shutdown
       ├─ Worker (src/common/worker/)  ← N normal workers + 1 main worker
       │    ├─ concurrentqueue message passing
       │    ├─ TimerManager (steady + system timers)
       │    └─ Lua Script (sol3)
       ├─ NetworkManager (singleton)  ← message dispatch, RPC registry
       │    ├─ Router → dispatches to workers by ServiceWorkerType
       │    ├─ gbnet Server/Listener/Session  ← TCP network layer
       │    ├─ RPC system (msgpack + protobuf + coroutine via async_simple)
       │    └─ IoServicePool → N io_service threads
       ├─ Protobuf (src/protobuf/)  ← generated .pb.h/.pb.cc
       ├─ Lua scripting (src/script/, res/script/main.lua)
       ├─ DB modules (src/db/mysql/, src/db/redis/)
       └─ HTTP client/server (src/network/http/)
```

**Key architectural facts:**
- There is **no standalone server library** — the root CMakeLists only builds test executables (`server_test`, `client_test`) and `tools/luasocket`. New server binaries are created as test subprojects that include all of `src/`.
- Each worker has its own Lua state (sol::state). Main worker runs the frame loop; normal workers process posted tasks.
- Router assigns messages to workers by `MessageType % 10000` → `ServiceWorkerType` (Normal=0, AI=1, Navigation=2).
- Graceful shutdown is 4 phases: stop IO → drain pending tasks → complete timer frame → cleanup.

## Code conventions

- **C++23**, `#pragma once` headers, `Microsoft`-style brace wrapping (custom clang-format)
- **Pointer alignment**: `int* a` (left)
- **Indent**: 4 spaces, no tabs
- **No column limit** (ColumnLimit: 0)
- **BinPackParameters: false** — one parameter per line in declarations
- **Namespace**: `gb` for project code, `NAMESPACE_BEGIN`/`NAMESPACE_END` macros
- **Singleton**: inherit `Singleton<T>`, call `T::Instance()` — lazy init, no thread safety on creation
- **Logging**: `LOG_INFO(...)`, `LOG_ERROR(...)` etc. via spdlog (variadic fmt-style)
- **Error handling**: return `0` for success, `-1` for failure. CHECK macros log but don't abort.
- **Test registration**: RPC handlers are registered inside worker threads via `w->Post([](){ NetworkManager::Instance()->Register(...); })`.

## Testing

- **Two test targets** exist: `server_test` and `client_test`.
- Both include **every source file** from `src/` via glob — no per-file selection.
- To add a new server binary, copy the `test/server_test/CMakeLists.txt` pattern and add it to root `CMakeLists.txt`.
- Run: after building, `start_process.bat -t Debug -n server_test.exe`.
- RPC registration happens inside `Worker::Post()` callbacks — **must be on the correct worker thread**.

## Protobuf codegen

```bash
cd res/proto
proto.bat   # generates descriptors.pb + src/protobuf/*.pb.h/.pb.cc
```

The bat file calls `tools/protoc.exe` twice: once for descriptor set, once for C++ output.

## Lua scripting

- `res/script/main.lua` is loaded at worker startup.
- Lua can register network handlers via `net.Listen(type, func, "ProtoName")` and RPC methods via `net.Register("method", func)`.
- Protobuf messages are accessible from Lua via the protobuf bridge (`create_msg`, field accessors).
- `start_debug.lua` enables LuaPanda debugger.

## Common gotchas

1. **Conan + Boost Cobalt bug**: Always use a profile with `boost/*:without_cobalt=True` and `tools.build:cxxflags+=["-DBOOST_CHARCONV_DISABLE_FLOAT128"]`.
2. **protobuf DLL on Windows**: `PROTOBUF_USE_DLLS` macro is required everywhere. The DLL copy logic is in `cmake/base.cmake` → `copy_dlls_to_target()`.
3. **VS2026 environment**: Visual Studio 2026 is not in PATH by default. Run from "Command Prompt for VS" or use the full vcvars path.
4. **CMake presets are Conan-generated**: `CMakeUserPresets.json` includes `build/Debug/generators/CMakePresets.json` — presets won't exist until after `conan install`.
5. **Lua state per worker**: Each worker has its own Script (sol::state). RPC/Listen registration in Lua is isolated to that worker.
6. **App entrypoint**: `main()` is in `test/server_test/main.cpp` (for server) — not in `src/`. The `App` base class provides the lifecycle skeleton.
7. **Linux build**: Clang is the primary Linux compiler (profiles/clang_debug_pr). GCC is also supported.
8. **Server config**: IP/port comes from `res/config/server_config.xml` and is selected by OS type (win_tcp vs linux_tcp).
