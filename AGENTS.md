# gb_server — C++ network game server framework

## Quick start

```bash
# Windows — run from "Command Prompt for VS" (VS2026 is not in PATH by default)
install_deps.bat
conan install . -pr=profiles/msvc_debug_pr --build=missing
cmake --preset conan-debug
cmake --build --preset conan-debug --parallel

# Linux — same flow, but always use a profile to work around Conan bugs:
conan install . -pr=profiles/clang_debug_pr --build=missing
cmake --preset conan-debug
cmake --build --preset conan-debug --parallel

# Run server from build output
start_process.bat -s login -t Debug
start_process.bat -s gateway -t Debug
start_process.bat -s scene -t Debug
# The executable expects -t <app_type> -r <res_path> at minimum
```

## Build system

### Build files

| Path | Purpose |
|---|---|
| `CMakeLists.txt` | Root — includes Conan toolchain, finds all deps, adds subdirectories |
| `cmake/base.cmake` | Utility functions: `collectHeadFiles`, `collectSrcFiles`, `collectAllSrcFiles`, `copy_dlls_to_target` |
| `cmake/def.cmake` | Build config: C++23, MSVC flags (`/utf-8`, `/EHa`, `/bigobj`), Linux/Clang/GCC flags, `conan_link_libraries` macro |
| `CMakeUserPresets.json` | Includes `build/Debug/generators/CMakePresets.json` (Conan-generated) |
| `conanfile.py` | Conan 2.0 recipe — lists all dependencies |

### Build system quirks

- **Conan-generated presets** — `CMakePresets.json` is generated by Conan into `build/<config>/generators/`. `CMakeUserPresets.json` includes them. **Never check in generated presets.**
- **protobuf uses DLLs on Windows** — `PROTOBUF_USE_DLLS` must be defined (handled per-target via `add_definitions(-DPROTOBUF_USE_DLLS)`).
- **C++23 required** — `CMAKE_CXX_STANDARD 23`, no compiler extensions. MSVC needs `/EHa` and `/utf-8` flags.
- **Build output layout**: `build/<config>/<config>/bin/` and `build/<config>/<config>/lib/`.
- **CMake presets are Conan-generated**: presets won't exist until after `conan install`.

### CMake target pattern

Every executable (test or server) follows the same pattern:
1. Include `cmake/base.cmake` and `cmake/def.cmake`
2. `collectAllSrcFiles` from `src/` (glob ALL .h/.cpp)
3. `collectAllSrcFiles` from `protobuf/` (generated code)
4. `collectSrcFiles` + `collectHeadFiles` from the target's own dir
5. `conan_link_libraries(target)` — links via the macro in `def.cmake`
6. `copy_dlls_to_target(target)` — copies DLLs on Windows

Binaries produced: `server_test`, `client_test`, `db_test`, `login_server`, `gateway_server`, `scene_server`.

## Architecture overview

```
test/server_test/main.cpp     ← test entrypoint (MyApp extends App)
test/client_test/main.cpp     ← test entrypoint (MyApp extends App)
server/login_server/main.cpp  ← login server (LoginApp extends ServerApp)
server/gateway_server/main.cpp← gateway server (GatewayApp extends ServerApp)
server/scene_server/main.cpp  ← scene server (SceneApp extends ServerApp)
  └─ App (src/app/)            ← frame loop, lifecycle hooks, graceful shutdown
       ├─ ServerApp (src/app/server_app.h)  ← base for game server processes
       │    ├─ CLI parsing (-t <type> -r <res> via cxxopts)
       │    ├─ Auto-creates Worker + Router registration
       │    ├─ Starts TCP Server (ip:port from server_config.xml by APP_TYPE)
       │    └─ OnServerInit() hook for subclasses
       ├─ Worker (src/worker/)  ← N normal workers + 1 main worker
       │    ├─ concurrentqueue message passing
       │    ├─ TimerManager (steady + system timers)
       │    ├─ Lua Script (sol3)
       │    ├─ WorkerExecutor / GbAsyncExecutor (async_simple adapter)
       │    └─ IWorkerLogic interface (OnStartup/OnUpdate/OnTick/OnCleanup)
       ├─ NetworkManager (singleton)  ← message dispatch, RPC registry
       │    ├─ Router → dispatches to workers by ServiceWorkerType
       │    ├─ gbnet Server/Listener/Session  ← TCP network layer
       │    ├─ RPC system (msgpack + protobuf + coroutine via async_simple)
       │    ├─ ThreadLocalRpcContext (per-thread RPC pending map)
       │    └─ IoServicePool → N io_service threads
       ├─ Protobuf (protobuf/)  ← generated .pb.h/.pb.cc from res/proto/*.proto
       ├─ Lua scripting (script/main.lua, src/script/)
       ├─ DB modules (src/db/redis/, src/db/postgres/)
       ├─ HTTP client/server (src/network/http/)
       ├─ Async (src/async/) — ShutdownManager, SignalHandler
       ├─ Timer (src/timer/) — Timer, TimerManager, SteadyTimer, SystemTimer
       └─ Base (src/base/) — Singleton, ResPath, MD5, util_string, timer_help
```

### Entrypoints (main.cpp)

Every executable follows the same pattern:
```cpp
int main(int argc, char* argv[]) {
    MyApp app(argc, argv);
    app.Init();
    app.Run();
    return 0;
}
```

### App type hierarchy

```
App (src/app/app.h)
 └─ ServerApp (src/app/server_app.h) — base for game servers
      ├─ LoginApp  (server/login_server/) — login server, includes HttpServer
      ├─ GatewayApp (server/gateway_server/) — gateway server
      └─ SceneApp  (server/scene_server/) — scene server
```

**ServerApp** handles CLI parsing (`-t <type_value> -r <res_path>`), auto-creates a NormalWorker, starts a TCP `gb::Server` on the IP/port from `res/config/server_config.xml` (looked up by `APP_TYPE`), and offers `OnServerInit()` for subclass hook.

Server presets in `start_process.bat`: login=8, scene=32, gateway=64.

**Key architectural facts:**
- All binaries include **every source file** from `src/` via glob — no per-file selection.
- There are **6 executables**: `server_test`, `client_test`, `db_test`, `login_server`, `gateway_server`, `scene_server`.
- Each worker has its own Lua state (sol::state). Main worker runs the frame loop; normal workers process posted tasks.
- Router assigns messages to workers by `MessageType % 10000` → `ServiceWorkerType` (Normal=0, AI=1, Navigation=2).
- Graceful shutdown is 4 phases: stop IO → drain pending tasks → complete timer frame → cleanup.
- App entrypoints: `test/server_test/main.cpp`, `test/client_test/main.cpp`, `server/*/main.cpp`.
- Server config: IP/port comes from `res/config/server_config.xml` and is selected by OS type (win_tcp vs linux_tcp) and `APP_TYPE`.

## Code conventions

- **C++23**, `#pragma once` headers, `Microsoft`-style brace wrapping (custom clang-format)
- **Pointer alignment**: `int* a` (left)
- **Indent**: 4 spaces, no tabs
- **No column limit** (ColumnLimit: 0)
- **BinPackParameters: false** — one parameter per line in declarations
- **Namespace**: `gb` for project code, `NAMESPACE_BEGIN`/`NAMESPACE_END` macros
- **Singleton**: inherit `Singleton<T>`, call `T::Instance()` — lazy init, no thread safety on creation (inline static C++17)
- **Logging**: `LOG_INFO(...)`, `LOG_ERROR(...)` etc. via spdlog (variadic fmt-style)
- **Error handling**: return `0` for success, `-1` for failure. CHECK macros log but don't abort.
- **RPC registration**: happens inside worker threads via `w->Post([](){ NetworkManager::Instance()->Register(...); })`.

## Lua scripting

- `script/main.lua` is loaded at worker startup via `require("main.lua")`.
- Each worker has its own Lua state (sol::state) with net, log, msgpack, protobuf bindings.
- Lua can register network handlers via `net.Listen(type, func, "ProtoName")` and RPC methods via `net.Register("method", func)`.
- Protobuf messages are accessible from Lua via the protobuf bridge (`create_msg`, field accessors).
- `script/start_debug.lua` enables LuaPanda debugger (127.0.0.1:8828).
- `script/test.lua` has RPC call examples.
- Scripts live at project root `script/` directory, NOT in `res/`.

## Thread model

```

│  │  - TimerManager::Update()（稳态 + 系统定时器）             │   │
│  │  - 执行队列任务 (ConcurrentQueue)                          │   │
│  │  - WorkerLogic::OnUpdate / OnTick                         │   │
│  └──────────────────────────────────────────────────────────┘   │
│                                                                 │
│  Normal Worker(s) — 独立的 thread, 内部帧循环                    │
│  ┌──────────────────────────────────────────────────────────┐   │
│  │  while (runing_):                                        │   │
│  │    wait_for(50ms) 或新任务到来                            │   │
│  │    ProcessFrame(elapsed)                                  │   │
│  └──────────────────────────────────────────────────────────┘   │
│                                                                 │
│  IO Thread Pool (IoServicePool)                                 │
│  ┌──────────────────────────────────────────────────────────┐   │
│  │  IoWorker × N                                             │   │
│  │  每个 IoWorker 运行自己的 io_context::run()                │   │
│  │  处理：accept、read、write、SSL handshake、心跳            │   │

│  └──────────────────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────────────────┘
```

## Message routing

```
Client → TCP → ServerImpl → Listener → Session
  │
  └─ on_received(buffer, meta_size, data_size)
       └─ Session::_received_callback
            └─ NetworkManager::OnReceiveCall
                 └─ NetworkManager::Dispatch
                      ├─ MsgMode::Msg      → FindListenFunction(type)
                      │                       → GetExecutor(type, id)
                      │                          ├─ Stateful：entity_id 精确绑定，未命中回退 hash
                      │                          └─ Stateless：纯 hash
                      │                       → WorkerExecutor::Dispatch → Worker::Post
                      │
                      ├─ MsgMode::Request   → FindRpcFunction(method)
                      │                       → GetExecutor(type, id)
                      │                          ├─ Stateful：entity_id 精确绑定，未命中回退 hash
                      │                          └─ Stateless：纯 hash
                      │                       → WorkerExecutor::Dispatch → Worker::Post
                      │
                      └─ MsgMode::Response  → WorkerIndex/LocalSeq从sequence解码
                                              → Worker::Post → TakePendingRpc(local_seq)
                                              → RpcCall::Done → 触发回调
```

### Meta message header

```cpp
struct Meta {
    MsgMode      mode{Msg};         // Msg / Request / Response
    uint64_t     id{0};             // user identifier (routing key)
    uint32_t     type{0};           // message type (Listen dispatch)
    CompressType compress_type{CompressTypeNone};
    uint64_t     method{0};         // RPC method name MD5 hash
    uint64_t     sequence{0};       // RPC sequence (encodes worker_index + local_seq)
};
```

### Router — ServiceWorkerType resolution

```
MessageType % 10000 → ServiceWorkerType
       0 % 10000 = 0  → Normal     (SWT_Normal = 0)
   10000 % 10000 = 0  → Normal
   10001 % 10000 = 1  → AI         (SWT_AI = 1)
   20000 % 10000 = 0  → Normal
   20001 % 10000 = 2  → Navigation (SWT_Navigation = 2)
```

**MessageType enum:**
```cpp
enum MessageType : uint32_t {
    MT_Login          = 0,       // → Normal
    MT_EnterScene     = 1,       // → Normal
    MT_AI_Run         = 10000,   // → Normal
    MT_AI_Skill       = 10001,   // → AI
    MT_AStar          = 20000,   // → Normal
    MT_LineFindPath   = 20001,   // → Navigation
};
```

**Worker selection (PickWorker):** default: `route_id % workers.size()`. Customizable via `SetWorkerIndexSelector`.

### HTTP routing (independent)

HttpServer uses **exact string path matching** via Boost.Beast. Three routing mechanisms:

| Feature | Msg | RPC (Request/Response) | HTTP |
|---|---|---|---|
| Route key | `meta.type` (uint32) | Request: `meta.method` (MD5) / Response: `meta.sequence` | URL path + HTTP method |
| Route table | `ListenMap` (type→func) | `RpcInterfaceMap` (method_hash→func) | `vector<RouteEntry>` |
| Worker dispatch | Router | Request: same as Msg / Response: direct by worker_index | IO thread direct |
| Serialization | protobuf | msgpack or protobuf | JSON/text/any |

## RPC system

RPC uses **msgpack** serialization (or protobuf for single-message params), **MD5 hash** (`MD5::MD5Hash64`) as method identifier. Supports sync callback and `CoRpc<T>` coroutine (async_simple).

### Key classes

- `RpcCall` — RPC request with timeout/cancel/callback
- `RpcReply` — RPC response (Invoke with msgpack pack)
- `CoRpc<T...>` — Coroutine wrapper (async_simple `Lazy<T>`)
- `WorkerExecutor` — Dispatch to worker threads
- `GbAsyncExecutor` — async_simple executor adapter
- `ThreadLocalRpcContext` — Per-thread RPC registration + pending call map (zero-lock)

### Sequence encoding

```cpp
union SequenceId {
    struct { uint64_t index : 32; uint64_t seq : 32; };
    uint64_t value;
};
// High 32 bits = worker_index, low 32 bits = local_seq
```

### Registration

```cpp
// C++
NetworkManager::Instance()->Register("method_name", [](RpcReply reply, int arg1, std::string arg2) {
    reply.Invoke(result);
});

// Lua
net.Register("method_name", function(reply, arg1, arg2)
    reply:Invoke(result)
end)
```

### Call

```cpp
// Callback style
auto call = std::make_shared<RpcCall>();
call->SetCallBack([](int result) { ... });
NetworkManager::Instance()->Call(call, "method_name", arg1, arg2);

// Coroutine style
auto result = co_await CoRpc<int>::execute(
    std::make_shared<RpcCall>(), "method_name", arg1, arg2);
```

## Shutdown (4-phase graceful)

```
Trigger: Ctrl+C or SignalHandler → App::Stop() → ShutdownManager::Shutdown()
  ├─ Phase 1: StoppingIO — IoServicePool::GracefulStop()
  ├─ Phase 2: CompletingTimers — TimerManager::EnterShutdownMode(), one final frame
  ├─ Phase 3: ProcessingTasks — Worker::EnterShutdownMode(), drain queues (5s timeout)
  └─ Phase 4: Cleaning — OnCleanup, join threads, OnUnInit, SignalHandler::Cleanup
```

## Frame loop

```cpp
// App::Run()
while (runding_) {
    if (SignalHandler::IsSignalReceived()) break;
    current_time = steady_clock::now();
    elapsed = current_time - last_time;
    last_time = current_time;
    OnUpdate(elapsed);
    main_worker->ProcessFrame(elapsed);  // TimerManager::Update + event queue + logic
    OnTick();
    // Frame rate control (default 60fps ≈ 16ms)
    sleep_for(frame_duration_ - frame_time);
}
```

## Error handling

- **Return values**: 0 = success, -1 = failure.
- **CHECK macros**: `CHECK(expr)`, `CHECK_EQ/NE/LT/LE/GT/GE` — log error on failure, don't abort.
- **Logging**: `LOG_INFO`, `LOG_DEBUG`, `LOG_WARN`, `LOG_ERROR`, `LOG_CRITI` — via spdlog.
- **GbLog**: spdlog wrapper supporting async mode, console/file output, log rotation.

## Timer system

Each Worker has its own `TimerManager` with **dual priority queues** (min-heaps):

- **SteadyTimer** — based on `steady_clock` (game logic timers)
- **SystemTimer** — based on `system_clock` (real-world time alignment)
- **Shutdown support**: `EnterShutdownMode()` cancels future timers, completes current frame

## Common gotchas

1. **Conan + Boost 1.90 bug**: Use a profile with `boost/*:without_cobalt=True` and `BOOST_CHARCONV_DISABLE_FLOAT128`.
2. **protobuf DLL on Windows**: `PROTOBUF_USE_DLLS` macro required per-target. DLL copy logic in `cmake/base.cmake`.
3. **VS2026 environment**: Visual Studio 2026 is not in PATH by default. Run from "Command Prompt for VS".
4. **CMake presets are Conan-generated**: presets won't exist until after `conan install`.
5. **Lua state per worker**: Each worker has its own Script (sol::state). Registration isolated.
6. **App entrypoint**: Multiple entrypoints exist: `test/*/main.cpp` and `server/*/main.cpp`.
7. **ServerApp** handles CLI parsing, worker creation, and TCP server start automatically.
8. **ResPath**: Resource root set via `-r <path>` CLI argument, defaults to `./res`.
